# AIHues v8 技术选型决策

> **版本**: 8.0 研发准备
> **日期**: 2026-05-20
> **状态**: 待决策

---

## 现状

v7.0 是一个 **纯前端、纯静态** 的 AI 工具站：

- 71 个 HTML 文件，约 52,000 行代码
- 57 个工具页面（开发工具 30 + 实用工具 16 + AI写作 11）
- 3 个小游戏
- 11 个功能页面
- 所有数据存在 localStorage（单机版）
- 部署在 GitHub Pages

---

## 核心矛盾

**用户想要的功能 v.s. 当前纯静态架构的限制**

| 想要的功能 | 纯静态能做？ | 需要后端？ |
|-----------|------------|----------|
| 用户注册登录 | ❌ 只能 mock（localStorage 模拟） | ✅ |
| 评分 & 评论持久化 | ❌ 换设备即丢失 | ✅ |
| 排行榜（跨用户） | ❌ 只能本地排行 | ✅ |
| Credit 系统（可赚可花） | ⚠️ 单机版可行但意义有限 | 建议上 |
| 许愿单投票 | ❌ 无法聚合跨用户需求 | ✅ |
| 成就系统 | ⚠️ localStorage 可行 | 可选 |
| 深色模式/i18n | ✅ 纯前端完全OK | ❌ |
| PWA 离线 | ✅ Service Worker 已支持 | ❌ |

**结论：如果要让产品真正上线运营，必须引入后端。**

---

## 方案对比

### 方案 A：继续纯静态（最小改动）

**架构**: 保持 71 个 HTML + CSS + JS，GitHub Pages 部署

**优点**:
- 零运维成本
- 无服务器、无数据库
- 加载极快（CDN 全球分发）
- 研发团队门槛低（只需前端）

**缺点**:
- 无法真正上线（无用户系统、无数据持久化）
- 评分/排行榜/许愿单投票无法跨设备
- Credit 系统只是单机体验
- SEO 受限于静态文件

**适合场景**: 个人作品展示、MVP 演示、不做用户运营的静态站

---

### 方案 B：Serverless 全栈（推荐）

**架构**: Next.js 14 (App Router) + Vercel + Vercel KV (Redis) + Vercel Postgres

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│   用户浏览器  │────▶│  Next.js 14  │────▶│  Vercel Postgres │
│  (React SPA)  │     │  (Serverless) │     │  (用户/评分数据)  │
└─────────────┘     └──────────────┘     └─────────────────┘
                            │
                            ▼
                     ┌──────────────┐
                     │  Vercel KV   │
                     │ (排行榜/缓存) │
                     └──────────────┘
```

**优点**:
- 团队只需写 React，无需关心服务器运维
- Vercel 自动 CI/CD，git push 即部署
- KV 处理排行榜高并发读取，Postgres 处理结构化数据
- Next.js 14 的 Server Actions 让 API 开发极简
- ISR (增量静态再生) 兼顾 SEO 和性能

**缺点**:
- Vercel 免费额度有限，用户多了要付费（$20/月起）
- 需要学习 Next.js App Router（学习曲线存在但平缓）
- 冷启动延迟（Serverless 通病，但 Next.js 有优化）

**预计工作量**: 2-3 人，2-3 周完成迁移 + 后端开发

**适合场景**: 要快速上线、快速迭代、团队 < 5 人

---

### 方案 C：经典全栈自建

**架构**: React/Vue 前端 + Node.js/Python 后端 + PostgreSQL + Redis + Nginx

**优点**:
- 完全掌控，不限平台
- 扩展性强，用户量大了随时扩容
- 不依赖第三方 PaaS 供应商
- 成本可控（云服务器 $5-20/月起）

**缺点**:
- 需要前端 + 后端 + 运维至少 3 人
- 部署、CI/CD、监控需要自己搭建
- 开发周期长（预计 4-6 周）

**适合场景**: 团队有后端能力、计划大规模运营、预算充裕

---

## 推荐决策

### 如果团队 ≤ 2 人、2 周内要上线 → 选方案 B (Next.js + Vercel)

理由：
1. 研发团队只需要写 React，一人就能搞定前后端
2. Vercel 的 Server Actions 让 API 开发只需写 TypeScript 函数，无需 REST API 设计
3. KV + Postgres 开箱即用，不需要数据库运维
4. 从 v7 的 HTML 迁移到 React 组件是一次性工作，完成后长期受益

### 如果团队 ≥ 3 人、有后端工程师 → 选方案 C

理由：
- 长期更可控、成本更低
- 数据完全自主
- 可以接入更多第三方服务

### 如果只是想展示作品、不做用户运营 → 选方案 A（保持现状）

## 技术栈详情（方案 B — 推荐）

### 前端
- **框架**: Next.js 14 (App Router, Server Actions)
- **UI**: Tailwind CSS + shadcn/ui（v7 已经在用 Tailwind）
- **状态管理**: Zustand（轻量，适合工具站）
- **国际化**: next-intl（替代当前 i18n.js）

### 后端
- **API**: Next.js Server Actions（零 API 路由样板代码）
- **ORM**: Prisma（类型安全的数据库操作）
- **数据库**: Vercel Postgres (PostgreSQL)
- **缓存**: Vercel KV (Redis，排行榜专用)
- **认证**: NextAuth.js（GitHub/Google OAuth + 邮箱登录）

### 部署
- **平台**: Vercel（免费额度：100GB 带宽/月，Serverless 函数 100GB-小时/月）
- **域名**: 自定义域名（Vercel 自动生成 SSL）
- **监控**: Vercel Analytics + Speed Insights

### 开发工具
- **语言**: TypeScript（全栈类型安全）
- **测试**: Vitest + Playwright
- **CI/CD**: GitHub Actions（Lint + Test + Deploy）

## 迁移策略（v7 HTML → v8 Next.js）

### Phase 1: 项目骨架（Week 1 Day 1-2）
1. 初始化 Next.js 14 项目（TypeScript + Tailwind）
2. 配置 Prisma + Vercel Postgres
3. 配置 NextAuth.js 认证
4. 配置 next-intl 国际化
5. 设计系统组件化（从 v7 CSS 提取公共组件）

### Phase 2: 核心页面迁移（Week 1 Day 3-5）
1. 首页（Hero + 分类导航 + 统计栏 + 小游戏入口）
2. 工具列表页（发现页）
3. 排行榜页
4. 许愿单页
5. 登录/注册页

### Phase 3: 57 个工具页面迁移（Week 2）
- 每个工具封装为 React 组件
- 保留纯前端逻辑（不依赖后端）
- 添加评分/评论 UI（数据存后端）

### Phase 4: 后端功能开发（Week 2-3）
- 用户系统（注册/登录/个人资料）
- 评分系统（提交/查询/聚合平均分）
- 排行榜（日榜/周榜/月榜）
- 许愿单投票
- Credit 经济系统
- 成就系统

### Phase 5: 游戏 & 功能完善（Week 3）
- 3 个小游戏接入用户系统
- 成就系统与后端联动
- 深色模式/i18n 全局生效
- PWA 支持（Next.js 内置）

### Phase 6: 测试 & 部署（Week 3-4）
- 单元测试覆盖核心工具
- E2E 测试覆盖用户流程
- Vercel 部署 + 自定义域名
- 性能优化（Lighthouse 90+）

## 不做清单（明确边界）

| 功能 | 原因 |
|------|------|
| 后端 Admin Dashboard | 初期用 Prisma Studio 直接看数据 |
| 图片压缩/OCR/PDF处理 | 需要后端处理大文件，超出 MVP 范围 |
| 实时多人游戏 | WebSocket 复杂度高，单机游戏足够 |
| 支付系统 | Credit 系统初期纯免费发放 |
| 复杂推荐算法 | 初期按分类 + 热度排序足够 |
